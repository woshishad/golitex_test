## Litex 是什么

Litex 是一个用来书写和检查数学证明的开源形式化语言。它的目标不是让数学证明变成一串技巧命令，而是尽量保留普通数学写作的形状：对象、条件、事实、计算链、集合成员、函数、存在、全称、分类讨论、归纳。

一句话说：

> 用户写事实，Litex 增长 verified context。

这不是语法糖，而是 Litex 的核心交互模型。用户写下一条数学事实；Litex 检查它是否能从当前上下文推出；如果通过，这条事实会进入上下文，后面的证明可以继续使用它、匹配它、实例化它、从它推出常规后果。

一个最小例子是：

```litex
forall x R:
    x = 2
    =>:
        x + 1 = 3
        x^2 = 4
```

这里用户没有先寻找 tactic 名字，也没有显式写“把 `x = 2` 代入再算术化简”。用户写的是目标事实本身；Litex 根据事实的形状、当前假设和内置的基础数学关系去检查它。

## 边界和免责声明

Litex 不是 Lean、Coq、Isabelle 的替代品，也不应该被宣传成已经成熟的
production formal verification 系统。它现在更准确的定位是一个研究型、实验型的
形式化语言：测试一种更小、更可读、fact-oriented 的界面，能不能让学生、领域科学家
和 AI agents 更便宜地产生有用的 checked mathematical data。

几个边界需要一直讲清楚：

| 边界 | 含义 |
| --- | --- |
| `know` | `know` 是显式假设或 proof debt，不是 Litex 证明出来的事实。 |
| builtin / infer rule | 这些规则属于 trusted mathematical background，需要文档、测试和可审计边界。 |
| 当前 examples | 240+ runnable examples 是有用的 prototype evidence，不是 production-ready 证明。 |
| 和 Lean 的关系 | Lean 有成熟 kernel、Mathlib、生态和社区；Litex 探索的是另一种较窄的 proof interface。 |
| AI 使用 | AI 生成的 Litex 代码仍然要看 verifier output、剩余 `know` 和失败原因，不能只看文本是否像证明。 |

所以更稳妥的说法是：Litex 不是“更快的 Lean”。它测试一个不同假设：

> 更小、更可读、以事实为中心的形式化语言，能否让许多普通数学推理更容易进入
> checked feedback loop。

这里的大 trusted base 不是偶然，也不是简单的历史包袱。Litex 选择的是
relation-first 的表面语言：成员关系、子集、等式、序、函数定义域、元组、有限集合、
用户定义谓词之间的普通数学关系，尽量由 checker 直接理解和复用。这样做的代价是
内核和 verifier 承担更多责任；收益是用户不用先处理很重的抽象层级、库定理名和
proof-command bookkeeping，可以直接写下一条数学事实，看它是否从当前上下文推出。

因此，正确的解释不是“Litex 已经达到了成熟小核系统的 soundness 形态”，而是：
Litex 在 prototype 阶段有意识地用更大的 trusted implementation，换取更自然、
更短的 checked feedback loop。这个取舍必须靠清楚的文档、测试、`verified_by`
输出、`know` 标注和 builtin / infer rule 边界来支撑。

## 最重要的特点

| 特点 | 含义 |
| --- | --- |
| fact-oriented proof interface | 证明由一条条数学事实增长，而不是主要由 tactic 脚本驱动。 |
| shape-directed matching | 用户写出目标事实的数学形状，checker 尝试匹配 builtin rules、known facts、known `forall`。 |
| explainable verifier feedback | 每条事实返回 `true` / `unknown` / `error`，成功时还能通过 `verified_by` 说明为什么成功。 |
| set-theoretic mathematical surface | 集合、函数、元组、成员关系、set-builder 都像普通数学对象一样出现。 |
| abstraction-first development | 可以先给出对象、关系和公理接口，再从这个抽象层开始推理。 |
| runnable textbook asset | The Mechanics of Litex Proof 已经包含 240+ runnable examples，覆盖计算、逻辑、归纳、集合、函数、关系等内容。 |

## 可以从合适的抽象层开始

很多数学工作不应该一开始就被迫展开到坐标、编码或构造细节。数学家经常先说明自己要研究的结构：有哪些对象、哪些关系、哪些公理，以及这些关系应该满足什么事实。

Litex 支持这种写法。一个文件可以用 `have` 引入基本域，用 `abstract_prop` 引入原始关系，用 `prop` 定义派生关系，用 `know` 记录公理或背景事实。后面的证明可以直接在这个抽象接口上增长；具体模型或更底层构造可以以后再补。

这类写法适合几何、公理化代数结构、商结构和分析接口：先把对象和关系说清楚，再把暂时作为公理或 proof debt 的部分逐步替换成 checked claims。

## 事实优先：证明就是上下文增长

Litex 脚本的基本单位是数学事实。事实一旦被验证，就成为后续证明可用的上下文。

```litex
forall a, b Q:
    a - b = 4
    a * b = 1
    =>:
        (a + b)^2 = (a - b)^2 + 4 * (a * b) = 4^2 + 4 * 1 = 20
```

这段证明读起来就是普通的代数计算链：

1. `(a + b)^2` 改写成 `(a - b)^2 + 4 * (a * b)`。
2. 用假设 `a - b = 4` 和 `a * b = 1` 代入。
3. 做数值计算得到 `20`。

Litex 的重点不是让用户手动标注每一步用了哪个 tactic，而是让计算链本身成为可检查的事实序列。

再看一个更短的例子：

```litex
forall r, s R:
    r + 2 * s = -1
    s = 3
    =>:
        r = (r + 2 * s) - 2 * s = -7
```

`r + 2 * s = -1` 和 `s = 3` 已经在局部上下文中。后面的等式链可以直接使用这些事实。

## 形状驱动匹配：先写数学形状，而不是先想 tactic 名字

Litex 鼓励用户先写“我现在需要的数学事实”。checker 会根据这条事实的形状去找证明路径：是否是基础算术、是否能由已知等式代入、是否能匹配一个已知 `forall`、是否是集合成员关系、是否是函数定义展开。

一个典型例子是全称事实的实例化：

```litex
have human nonempty_set, Socrates human
abstract_prop mortal(x)

know forall x human:
    $mortal(x)

$mortal(Socrates)
```

最后一行 `$mortal(Socrates)` 的形状和已知事实 `forall x human: $mortal(x)` 匹配。Litex 看到 `Socrates human`，于是可以把 `x` 实例化为 `Socrates`。

这体现了 Litex 的一个关键使用方式：`forall` 不是静态装饰，而是可以被后续目标事实按形状调用的数学规则。

## 可解释反馈：true / unknown / error

Litex 的反馈不只是“过了”或“没过”。每条事实都有三类结果：

| 结果 | 含义 | 常见下一步 |
| --- | --- | --- |
| `true` | Litex 找到了证明路径。 | 保留该事实，继续下一步。 |
| `unknown` | 语句有意义，但当前上下文不足以证明。 | 补一个更小的中间事实、条件、见证、分类讨论或 `forall`。 |
| `error` | 语法、作用域或定义良好性失败。 | 修正未声明对象、函数定义域、除零、语法结构等问题。 |

成功时，输出还可以说明成功原因。例如下面这种 `verified_by` 信息表示该事实是通过引用一个已知 `forall` 得到的：

```json
{
  "result": "success",
  "type": "AtomicFact",
  "stmt": "$mortal(Socrates)",
  "verified_by": {
    "type": "cite forall fact",
    "cited_stmt": "forall x human:\n    $mortal(~1x)"
  }
}
```

这对局部修复很重要。用户不只知道“这一行成功”，还知道它是通过 builtin rule、known fact、known `forall`，还是其他证明路径成功的。失败时，用户也能区分是数学 gap 还是语法/定义良好性问题。

## 集合论风格的数学表面

Litex 的表面语言接近很多教材里默认使用的集合论数学对象：数集、集合、元素、函数、元组、笛卡尔积、set-builder、有限集合。

### 成员关系和 set-builder

```litex
1 $in {n Z : n <= 3}
2 $in {1, 2, 3}
not 4 $in {1, 2, 3}
```

`{n Z : n <= 3}` 就是“整数 `n` 中满足 `n <= 3` 的集合”。证明 `1 $in {n Z : n <= 3}` 时，Litex 检查的是把 `1` 代入条件后的事实 `1 <= 3`。

有限集合也按普通数学方式使用：

```litex
forall a {1, 2, 3}:
    a = 1 or a = 2 or a = 3
```

### 函数和定义域

```litex
have fn g(x R: x > 0) R = x + 1

g(1) = 2
```

函数 `g` 的输入域写在参数位置：`x R: x > 0`。调用 `g(1)` 时，Litex 会检查 `1` 满足这个输入条件。

分段函数也可以按数学形状写：

```litex
have fn h(x R) R by cases:
    case x = 2: 3
    case x != 2: 4

h(2) = 3
```

### 集合外延性

集合相等通常靠互相包含来证明。Litex 也支持这种 element-chasing 风格：

```litex
by extension union({1, 2}, {2, 4}) = {1, 2, 4}:
    claim forall! x union({1, 2}, {2, 4}) => {x $in {1, 2, 4}}:
        by cases x $in {1, 2, 4}:
            case x $in {1, 2}:
                by enumerate finite_set:
                    prove:
                        forall a {1, 2}:
                            a $in {1, 2, 4}
            case x $in {2, 4}:
                by enumerate finite_set:
                    prove:
                        forall b {2, 4}:
                            b $in {1, 2, 4}

    by enumerate finite_set:
        prove:
            forall x {1, 2, 4}:
                x $in union({1, 2}, {2, 4})
```

这个例子展示的是普通集合证明的形状：左到右、右到左、分类讨论、成员关系传递。

## Litex 和 Lean 的简要对比

Litex 不是“更快的 Lean”，也不是 Lean 的替代品。Lean 是成熟 theorem prover，有强大的类型论基础、大规模 Mathlib、活跃社区和丰富工具链。Litex 目前更年轻、更实验，目标也更窄：让很多教材级和应用级数学推理用接近普通数学写作的方式进入可检查反馈循环。

| 维度 | Litex | Lean |
| --- | --- | --- |
| 默认交互 | 写下一条数学事实，让 checker 验证。 | 写 theorem/proof term/tactic，显式引导证明状态。 |
| 用户主要记忆内容 | 数学形状：等式链、成员关系、见证、分类、归纳、`forall`。 | tactic 名字、库定理名、类型结构、proof state 操作。 |
| 上下文模型 | 已验证事实持续增长，后续事实可匹配和复用。 | hypothesis、local context、goal state 由证明脚本管理。 |
| 表面风格 | 集合、函数、元素、set-builder 更接近普通数学文本。 | 更精确、更通用，但用户更早接触类型、subtype、coercion、library convention。 |
| 反馈重点 | `true` / `unknown` / `error` 和 `verified_by`，适合局部修复。 | tactic state、error message、elaboration feedback，适合通用形式化开发。 |
| 当前优势 | 低摩擦、事实驱动、局部反馈、教材 proof loop 清楚。 | 库深度、生态成熟度、soundness 传统、社区和工具链。 |

## 局部反馈循环

做较大证明时，最困难的常常不是一次性写出一个完美证明，而是知道局部失败在哪里、下一步应该补什么。Litex 的交互正好是局部的：

1. 写自然语言证明计划。
2. 把计划拆成 Litex 事实。
3. 运行 checker。
4. 对 `error` 修语法、作用域、定义域、良定义性。
5. 对 `unknown` 补更小的中间事实、见证、分类讨论或 `forall`。
6. 对 `true` 保留事实，并利用 `verified_by` 理解成功路径。
7. 重复，直到证明中的 gap 足够小或完全消失。

这种动作比盲目搜索 tactic 或 lemma 名字更直接：写下一条数学事实 -> 运行 -> 看反馈 -> 缩小 gap。

一个函数证明示例：

```litex
prop injective_fn(S, T set, f fn(x S) T):
    forall x1, x2 S:
        f(x1) = f(x2)
        =>:
            x1 = x2

prop surjective_fn(S, T set, f fn(x S) T):
    forall y T:
        exist x S st {y = f(x)}

prop bijective_fn(S, T set, f fn(x S) T):
    $injective_fn(S, T, f)
    $surjective_fn(S, T, f)

have fn p_affine(x R) R = 2 * x - 5

claim:
    prove:
        forall x1, x2 R:
            p_affine(x1) = p_affine(x2)
            =>:
                x1 = x2

    x1 = ((2 * x1 - 5) + 5) / 2 = (p_affine(x1) + 5) / 2 = (p_affine(x2) + 5) / 2 = ((2 * x2 - 5) + 5) / 2 = x2

$injective_fn(R, R, p_affine)

claim:
    prove:
        forall y R:
            exist x R st {y = p_affine(x)}

    witness exist x R st {y = p_affine(x)} from (y + 5) / 2:
        y = 2 * ((y + 5) / 2) - 5 = p_affine((y + 5) / 2)

$surjective_fn(R, R, p_affine)

$bijective_fn(R, R, p_affine)
```

这个证明很适合局部修复：如果 injective 部分失败，就补等式链；如果 surjective 部分失败，就检查 witness；如果函数调用失败，就检查定义域。

## The Mechanics of Litex Proof 

Litex 目前最硬的资产不是一个 toy README example，而是一批可以运行、可以回归测试的 examples、标准库片段和数据集翻译尝试。

这套材料把 Heather Macbeth 的 The Mechanics of Proof 中的Lean代码改写成 Litex proof style，已经形成连续章节和大量可运行片段。它的价值在于：读者不是只看一个漂亮例子，而是可以看到 Litex 如何处理一整套基础数学训练。

| 章节 | 内容 |
| --- | --- |
| Chapter 1 | 等式、不等式、计算链、代入和 verifier output。 |
| Chapter 2 | 全称、蕴含、局部 claim、存在见证、反证、分类讨论。 |
| Chapter 3 | 奇偶性、整除、模运算、可复用谓词和见证。 |
| Chapter 4 | 更复杂的结构化证明、唯一性、iff、反例和多重假设。 |
| Chapter 5 | 逻辑结构，`and` / `or` / `not` / implication-shaped facts。 |
| Chapter 6 | 归纳、强归纳、递归序列、递归函数和下降度量。 |
| Chapter 7 | 数论，包含欧几里得素数无限性一类证明结构。 |
| Chapter 8 | 函数、单射、满射、双射、反函数，以及较大的函数综合证明。 |
| Chapter 9 | 集合、成员、子集、外延性、并交补、幂集和 Cantor-style diagonal proof。 |
| Chapter 10 | 关系、基数、有限/无限、可数性、双射关系。 |

更重要的是，这套材料不是孤立展示某个 theorem，而是一整条
verifier pressure 下的开发路径。在 Codex 协助下，Litex 用约一周时间
形成了整本 Mechanics 的可运行验证材料；MATH500 和 MiniF2F-style
任务则继续作为压力测试，帮助发现缺少的定义、lemma、builtin rule、
infer rule、语法和诊断能力。成功的题目沉淀成 example 或 benchmark；
失败的题目变成明确的 gap，而不是只留下“模型不会做”的结论。

## 现在应该怎样理解 Litex

Litex 目前仍是实验性系统，不应该被描述成已经成熟的 production theorem prover，也不应该被说成已经能替代 Lean、Coq、Isabelle。

更准确的理解是：Litex 正在探索一种不同的形式化接口。它把普通数学写法尽量保留下来，让事实逐步进入 verified context，并把 checker 的成功和失败解释成可以被人使用的反馈。

当前最值得关注的不是“它是否已经完成了一切”，而是它已经把一个重要问题做成了可运行、可测试、可迭代的工程对象：

> 如何让普通数学推理进入机器可检查、可解释、可反馈的循环。

## 结语

Litex 的核心不是“更短的语法”，而是一个更直接的数学反馈界面：用户写事实，checker 按形状匹配和验证，成功的事实进入上下文，失败的事实给出局部修复信号。

这对学习和调试形式化证明有用。因为用户最需要的不是一句模糊的失败，而是明确知道：这一行是 `true`、`unknown` 还是 `error`；如果成功，是被什么证明的；如果失败，下一步应该缩小哪个数学 gap。
